Object Oriented UX

I approached Matcha using the concept I picked up from a medium article, Object-Oriented UX inspired by programming languages.

I started by listing the objects involved in organising a match event:

  • Players
  • Courts
  • Matches
  • Score

Then, I defined each object‚Äôs attributes ‚Äî basically, what ‚Äúinfo‚Äù each object needs to carry.

  • Players have a name and status (playing or resting).
  • Courts have a name and a status (empty or occupied).
  • A Match is made up of players, a court, score and status (playing or finished).
  • The score records who won the match and the final points.

After that, I mapped out how these objects connect to each other:

  • A Match needs players, a court and score assigned to it.
  • Courts become available or occupied depending on Match status.
  • Players can be actively playing or resting

I built everything around the real-world relationships between these objects instead of thinking screen-by-screen.

That way, the user flow naturally mirrors the mental model of a tired host ‚Äî not forcing them to ‚Äúthink like an app‚Äù but letting them just think about players, matches, and courts like they already do in real life.

The Flow

The interface follows the principle of progressive disclosure ‚Äî inspired by tools like Typeform. Matcha‚Äôs match setup flow is built to feel like a calm onboarding ‚Äî step-by-step, no clutter, no cognitive overload. It keeps the setup easy and focused.

The sequence is intentional:

  1. Select match type ‚Äî Singles, Fixed Doubles, or Random Doubles
  2. Insert player names
  3. Set the winning points ‚Äî 5, 7, 15, 21, or custom
  4. Insert the courts
  5. Generate match

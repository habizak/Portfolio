Matcha

The Gist

I was at a DUPR match (even though I suck at Pickleball) and the host was struggling to organize the games. More specifically — they couldn’t figure out who should play next.

The app (Re:Club) listed all the matches, but that’s about it. No filters. No court assignments. No way to tell who’s currently playing or which courts were free. Sure, the app lets you export the match list, and the host even printed it out for easier reference. But that wasn’t enough to stop the chaos.

Time was ticking. People were getting tired. Things were getting frantic. And all I wanted to say was:

“There’s an app for that.”
But there wasn’t.
Yet.

The Design Process

This app needs a place to live. The choice was between a mobile app or a website.

I picked the latter — for two simple reasons:

  • I only know HTML, CSS, and some JavaScript 🥲
  • A website is way easier to scale and works on any device, no messy app store submissions involved.

For the brand, I kept it simple: I called it Matcha — because Match Generator > Match > Matcha. From there, I had a color direction right away.

For the font, I wanted something circular and friendly. I knew I wanted to stay within Google Fonts (but not Inter — Inter is too overused, sorry not sorry).

I chose **** (TBD) for its soft, rounded feel. The UI follows that same softness — lots of rounded shapes to make it look welcoming and simple.

Design Process (Continued)

I approached Match Robin using the concept of object-oriented design. I started by listing the “objects” the app would need to handle:

  • Number of Players
  • Players
  • Match Type
  • Winning Points
  • Matches
  • Number of Courts
  • Courts
  • Score

The interface follows the principle of progressive disclosure — inspired by tools like Typeform. Users only see one step at a time, with just enough context to move forward. It keeps the setup calm, easy, and focused. And yes — users can move backward sequentially to update earlier details without confusion.

The Flow

I once attended a design event where a speaker said something that stuck with me:

“Graphic designers who move into UI/UX often only design for the happy path.”

That hit hard. Because I didn’t want Match Robin to fall into that trap.

I’ve always believed that the developer must be involved as early as possible in the design process. This prevents developer deaths by up to 42% due to shock (a statistic that I just made up).

The truth is — designers and developers think differently. But that difference? It creates better solutions, just like how the rotary phone dial inspired the iPod clickwheel.

Problems Caught

  • Odd number of players → Luna suggested a “bye system” to rotate players fairly
  • Local Storage structure → We needed a clear system to save setup and results without database complexity
  • Court/player imbalance → What happens when there are too few players and too many courts — or vice versa?
  • Match type limitations → Not all match types work with all player counts (e.g. random doubles with 7 players)
  • Post-generation editing → Should users be allowed to change matches after generation? (Spoiler: no)

Match Setup

Match Robin’s setup flow is built to feel like a calm onboarding — step-by-step, no clutter, no cognitive overload.

The sequence is intentional:

  • Insert player names
  • Select match type — Singles, Fixed Doubles, or Random Doubles
  • Set the winning points — 5, 7, 15, 21, or custom
  • Name your courts
  • Review everything
  • Generate matches

Each step reveals just enough info to keep things moving — a progressive disclosure approach that keeps hosts focused without overwhelming them.

Behind the scenes, the app handles things like:

  • Ensuring match type works with number of players
  • Handling odd numbers with a “bye” system
  • Preventing over-assignment of courts
  • Saving everything in localStorage — no accounts, no servers, just straight into action

Match Display

Once matches are generated, Match Robin shifts into game mode.

The match display screen shows exactly what the host needs to know — nothing more, nothing less:

  • Empty courts — visually marked and ready to be filled
  • Current matches — with court names and player assignments
  • Previous matches — with scores and court history
  • Rested players — so hosts know who’s up next

The host can assign the next match by picking from a list of valid, unplayed matchups. Matches are filtered automatically based on court availability and player status.

When a match ends, the court is marked empty and becomes available for the next matchup. The round ends when all scheduled matches have been played — or when everyone’s too tired to continue.
